ESTRUCTURA DE INICIO (igual en todas):
  getContent(patient.levelId)                    → datos del nivel
  getDifficultyForActivity(id, componentLevels)  → 'inicial'|'intermedio'|'avanzado'
  data.key[difficulty] ?? data.key.inicial ?? [] → array de ítems
  estimulusSettings.exerciseCount[id] ?? 12      → n° ejercicios (default 12)
  items.slice(0, n)                              → limitar cantidad

ESTADO COMÚN:
  idx, score, selected, feedback, answered, showNext — useState
  nextAction — useRef (closure para avanzar/finalizar)
  current = items[idx]

CICLO DE RESPUESTA (todas):
  handleAnswer → setAnswered(true) → playFeedback(correct|wrong, animationsEnabled)
  → setTimeout(1000) → setFeedback({type, text})
  → setTimeout(800)  → setShowNext(true)
  → botón "Siguiente →" → nextAction.current()
  → idx+1 < total: setIdx(idx+1) | idx+1 === total: onFinish(score, total)

ESTIMULUESSETTINGS RESPETADOS EN TODAS:
  animationsEnabled     → .no-anim class + playFeedback param
  backgroundElements    → whiteBg: background 'white'
  largerText            → instrSize (13–18px según pantalla)
  simplifiedInstructions→ texto de instrucción abreviado
  reducedOptions        → max 2 opciones (excepto MinimalPairs: siempre 2)
  exerciseCount[id]     → n° de ejercicios

ESTIMULUESSETTINGS PARCIALES (solo algunas):
  simultaneousAudioVisual → MinimalPairs (rate 0.7 vs 0.82), BuildWord (auto-play),
                             Listen (instrucción + rate + delay)
  sequentialStimulus      → Listen (retrasa habilitación del botón audio 2000ms)
  slideTransitionDelay    → DECLARADO en MinimalPairs, BuildWord, Listen, Narrative
                             pero NUNCA USADO (variable muerta en 4 pantallas)

PROPS COMUNES:
  onFinish(score, total), onBack

CLASES CSS USADAS POR TODAS:
  .screen, .activity-header, .back-btn, .activity-title,
  .game-area, .instruction, .feedback-banner.correct/wrong, .check-btn

5. DEUDAS TÉCNICAS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  [DEUDA-1] HomeScreen — progressPct hardcodeado
    72%, 60%, 40%, 55% son estáticos en LANGUAGE_LEVELS.
    No reflejan el historial real del paciente.

  [DEUDA-2] CSS no usado en index.css
    .activities-grid, .activity-card, .activity-card.teal/purple/peach/yellow,
    .dot, .progress-dots, .progress-fill.teal/purple/yellow/coral
    Estas clases existen en CSS pero ninguna pantalla las usa actualmente.
    HomeScreen usa botones con 100% inline styles en vez de .activity-card.

  [DEUDA-3] speak() duplicado en 4 pantallas
    MinimalPairsScreen, BuildWordScreen, ListenScreen, SemanticScreen definen
    cada una su propia función speak() module-level con parámetros ligeramente
    diferentes (MinimalPairs tiene pitch explícito, las otras no).
    Podría extraerse a utils/speech.js. No es un bug pero es duplicación.

  [DEUDA-4] Inconsistencia pitch en speak()
    MinimalPairs: speak(text, rate, pitch) — 3 parámetros, u.pitch = pitch
    BuildWord/Listen/Semantic: speak(text, rate) — 2 parámetros, u.pitch = 1.05 hardcodeado
    En la práctica el valor es el mismo (1.05) pero la firma es distinta.

  [DEUDA-5] SemanticScreen — formato options de definitions no uniforme
    Línea 64: o === current.data.correct || o.word === current.data.correct
    Acepta tanto string[] como {word,emoji}[] porque el formato varía entre niveles.
    Debería normalizarse en los JSON o en getContent().

  [DEUDA-6] ListenScreen — speak() con texto, no audio real
    current.sound es texto que se lee con TTS (SpeechSynthesis).
    No hay archivos .mp3 ni Web Audio API. Aceptable para MVP pero limitado
    para sonidos no-verbales (animales, instrumentos) a futuro.
